MVP 边界：V1 到底包含什么？

你的文档在两处提到了 MVP，但有一个张力需要对齐

⚡ 核心张力

文档说 MVP "不需要复杂 AI 分析"，但同时描述了 AI 是"零阻力录入体验"的关键——没有 AI 自动分类，用户在 solo capture 时就必须手动打 Pillar 标签，这会产生摩擦。

V1 的分类机制，你倾向哪种？

A

规则驱动（无 AI）

来源决定 Pillar：GitHub → Output，Stripe → Financial，CLI 手动指定 -p input

优点

  • V1 零 AI 依赖
  • 100% 可预测

缺点

  • CLI 手动需带参数
  • 无法处理模糊输入
B

AI 异步分类（轻 AI）

先裸存 Entry，后台 Claude API 自动打 Pillar 标签，错误的在 Weekly Review 纠正

优点

  • CLI 真正零摩擦
  • 体现产品核心差异

缺点

  • V1 需接 AI API
  • 分类有延迟
C

混合：规则优先 + AI 兜底

已知来源（GitHub/Stripe）走规则；未知来源的 CLI 输入走 AI 分类

优点

  • 覆盖 80% 场景无需 AI
  • 体验最完整

缺点

  • 复杂度略高
  • 需要同时维护两套逻辑
我的建议

选 C（混合）。GitHub Commit 和 Stripe 交易天然有来源语义，规则覆盖最高频场景；CLI 手打的碎片化想法才是 AI 分类的价值所在。两套逻辑共存，但工程复杂度可控。
